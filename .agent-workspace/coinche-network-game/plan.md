# Implementation Plan - Networked Terminal Coinche Game

This plan outlines the systematic implementation of a networked, terminal-based
Coinche (belote coinchée) card game: an asyncio TCP server hosting multiple
4-player tables, and a `rich`-based CLI client that joins a table by IP/host,
port, table key, and player name, then plays a full game (deal → bid → trick
play → score → repeat until target score reached).

## Assumptions Requiring Approval

These resolve every `[VARIANT]`/Open Question flagged in
[context-map.md](context-map.md). **Please review and challenge any of these
before Tony Builder starts implementation** — each is a concrete, coded

| A7 | Tout-Atout point table | All 4 suits use the **trump ladder** (J=20, 9=14, A=11, 10=10, K=4, Q=3, 8=0, 7=0 → 62/suit). No bonus for an *undeclared/incidental* capot (winning all 8 tricks without having bid Capot) — normal tally applies in that case. | Common flat value; some clubs use 400/500 — pick one number. |
| A9 | Coinche / Surcoinche multiplier | **Coinche = ×2**, **Surcoinche = ×4**, applied to the hand's final scored amount (not to the raw card points). Only the bidding team may surcoinche, and only once (no further re-double). | Matches context-map.md's default suggestion. |
| A10 | Contract-failed scoring | If attacking team's captured points (incl. belote/rebelote, dix-de-der) < bid amount ("chute"): attacking team scores **0**, defending team is credited the **entire available point pool for that declaration** (162 for a normal one-suit-trump contract, 258 for Tout-Atout, per A7), independent of what they actually captured. Belote bonus is always credited to whoever holds it, win or lose. | Directly from context-map.md, not really a variant. |
| A11 | Belote/Rebelote announcement | **Auto-detected and credited by the server** — no explicit "announce" client action. If a player holds both K+Q of the trump suit and plays both over the round, the +20 bonus is automatically credited to their team. | Simplifies protocol (no `announce_belote` message type); real club play requires a verbal announcement and forfeits the bonus if forgotten — this simplification always awards it. Flag if strict announcement enforcement is wanted. |
| A12 | Target score / game end | **1000 points** cumulative, first team to reach or exceed it at the end of a hand wins; configurable via `--target-score` CLI flag on the server (default 1000). | Common club value; 1500/2000 also common. |
| A13 | Simultaneous target reached / tie | If, after the deciding hand, both teams are ≥ target and **tied**, one more hand is played (sudden death) rather than ending in a draw. | A14 | Table key semantics | **Host-chosen, not server-generated**: whichever player first sends `join` with a given table key implicitly creates that table (lazily, in-memory). Reject `join` if the table already has 4 seated players, or if a game is already in progress on that table **and the joining name doesn't match a currently-disconnected seat (see A16)**. | Matches "simple CLI" scope; no persistence/auth requested. |
| A15 | Player-name collisions | Reject `join` with an `error` (`NAME_TAKEN`) if the requested name (case-insensitive) is already seated **and currently connected** at that table. If the name instead matches a currently-*disconnected* seat at a table with a game in progress, the join is treated as a **reconnection** (A16) rather than rejected. | Needed for unambiguous per-player addressing; extended to distinguish "name taken" from "this is a reconnect" per the new reconnection policy. |
| A16 | Reconnection policy (mid-game disconnect) | **Replaces the previous "disconnect ends the table" policy.** The server is the sole source of truth: it keeps a table's full in-memory game state (hand, current trick, bidding history, scores, whose turn) across a disconnect. When a seated player's socket drops **after the game has started**, the server marks that seat's `ClientSession` as `disconnected` (not removed) and pauses on that seat's turn if it comes up — it does **not** end the table, and does **not** auto-play/forfeit for them (no bot, no timeout). The server broadcasts a new `connection_status` message (`{seat, name, status: "disconnected"}`) so the other clients see a "waiting for `<name>` to reconnect" status rather than the game silently hanging. A new `join` with the same `table_key` + `player_name` (case-insensitive) matching that disconnected seat is treated as a **reconnection**: the server re-attaches the new `StreamWriter` to that seat and sends a new `resync` message (full state snapshot: hand, current trick/bidding state, whose turn, cumulative scores) instead of the normal `joined`/`deal` flow, then broadcasts `connection_status` `{status: "reconnected"}`. Dropping **before** the game starts (still in the waiting room) is unchanged from before: it simply frees the seat and broadcasts `lobby_update` — no reconnection semantics apply since no game state exists yet. The **client** independently auto-retries connecting (simple backoff loop, a few attempts with increasing delay) if its socket drops; if the client *process itself* crashes/exits, that session is simply lost — no persistence/resume beyond the process's own lifetime. | Explicit user directive, now in scope (was deferred in discoveries.md DISC-003). Deliberately minimal: no bot takeover, no timeout/forfeit, no cross-process/server-restart persistence — see Pre-flight Devil's Advocate Review for the accepted limitations this leaves open. |
| A17 | Chat / passthrough messages | A minimal `chat` message type is included (client → server → broadcast to table), free-text, no persistence, no rate limiting beyond the general line-length cap. | Requested by the protocol-definition ask ("chat/passthrough if any"); trivial to implement, not a rules variant. |

## Network Protocol Definition

Transport: **newline-delimited JSON ("JSON Lines") over asyncio TCP streams**.
Every message is one JSON object of shape `{"type": "<msg_type>", "payload": {...}}`
followed by `\n`, UTF-8 encoded. `asyncio.start_server`/`open_connection`'s
default `StreamReader` line-length limit (64 KiB) is relied on as a built-in
guard against oversized/malformed input (OWASP-relevant DoS mitigation — no
custom framing needed).

Card wire format: strings matching [demo_table.py](../../demo_table.py)'s
convention, e.g. `"10♥"`, `"V♠"`, `"D♦"`, `"R♣"`, `"A♠"` (ranks `7,8,9,10,V,D,R,A`;
suits `♠ ♥ ♦ ♣`). Seats are `"N" | "E" | "S" | "W"`. Teams are objective,
seat-based ids (`"NS"` / `"EW"`) — **not** player-relative "nous/eux"; the
client maps its own team to "nous" (cyan) and the other to "eux" (magenta)
locally, per A2.

### Client → Server messages
- `join` `{table_key: str, player_name: str}` — also doubles as a **reconnect** request: if `player_name` (case-insensitive) matches a currently-disconnected seat at a table with a game in progress, the server treats it as a reconnection instead of a fresh seat request (A16).
- `bid` `{action: "pass" | "bid" | "coinche" | "surcoinche", trump?: "♠"|"♥"|"♦"|"♣"|"tout_atout", points?: int|"capot"}` — **no `"sans_atout"` value exists**; it is not a legal declaration in this game.
- `play_card` `{card: str}`
- `chat` `{text: str}`

### Server → Client messages
- `joined` (to the joining client only) `{table_key, seat, players: [{seat, name}], target_score}`
- `lobby_update` (broadcast) `{players: [{seat, name}], seats_filled: int, waiting_for: int}`
- `deal` (private, one per player) `{hand: [card...], dealer_seat, first_bidder_seat, round_number}`
- `bid_request` (to the player to act) `{current_highest_bid: {...}|null, legal_actions: [...], can_coinche: bool, can_surcoinche: bool}`
- `bid_update` (broadcast) `{seat, action, trump?, points?, next_to_act}`
- `bidding_result` (broadcast) `{outcome: "contract"|"redeal", attacking_team?, seat?, trump?, points?|"capot", coinche_level?: 1|2|4}`
- `play_request` (to the player to act) `{legal_cards: [card...], current_trick: [{seat, card}], trump}`
- `card_played` (broadcast) `{seat, card, current_trick: [{seat, card}]}`
- `trick_result` (broadcast) `{winner_seat, trick: [{seat, card}], points_won, tricks_played, tricks_remaining}`
- `round_score` (broadcast) `{team_NS: {card_points, contract_result, belote_bonus, capot_bonus, multiplier, total}, team_EW: {...}, cumulative: {NS, EW}, next_dealer_seat}`
- `game_over` (broadcast) `{final_scores: {NS, EW}, winning_team}`
- `resync` (private, to the reconnecting client only, new — A16) `{table_key, seat, hand: [card...], phase: "bidding"|"trick_play", current_highest_bid: {...}|null, bid_history: [...], current_trick: [{seat, card}], trump, whose_turn: seat, cumulative_scores: {NS, EW}, round_number, dealer_seat}` — sent instead of the normal `joined`/`deal` flow when a `join` matches a disconnected seat mid-game; built by `Game.snapshot_for(seat)`.
- `connection_status` (broadcast, new — A16) `{seat, name, status: "disconnected"|"reconnected"}` — announces a mid-game disconnect ("waiting for `<name>` to reconnect") or a successful reconnection; clients render this as a persistent status banner, especially useful when `whose_turn` is the disconnected seat so the game doesn't appear to silently hang.
- `error` (to offending client) `{code, message}` — codes: `NOT_YOUR_TURN, ILLEGAL_BID, ILLEGAL_CARD, TABLE_FULL, GAME_IN_PROGRESS, NAME_TAKEN, MALFORMED_MESSAGE`
- `chat` (broadcast) `{seat, text}`

## Security & Anti-Cheat Model (Server Authority)
- **The server is the sole authority on turn order, legal bids, and legal card plays.** Clients only ever send *intent* messages (`bid`, `play_card`) describing what the player wants to do — they never send resolved/pre-computed state.
- Every incoming `bid`/`play_card` is independently re-validated by the server against the live `Game`/`RoundState` in `game.py` (whose-turn check, `rules.legal_bid_actions`/`is_valid_bid`, `rules.legal_cards_to_play`) before it is applied to state or broadcast to anyone. An out-of-turn or illegal action is rejected with the existing `error` message (`NOT_YOUR_TURN`, `ILLEGAL_BID`, `ILLEGAL_CARD`) and the sender's local state does not change.
- **Client-side legality display is a UX convenience only, never trusted.** The keyboard-shortcut numbered menus built by `ui.py` (only showing currently-legal bids/cards, see below) exist purely to make the CLI usable — a modified/malicious client sending an illegal or out-of-turn message is rejected exactly the same way a legitimate client would be if its local view were stale.
- No client ever receives another seat's private hand; `deal`/`resync` payloads are always addressed to a single connection via `send_to` (never `broadcast`) and contain only that seat's own cards.

## Concurrency Model
- Single asyncio event loop; `asyncio.start_server` accepts connections, one `handle_connection(reader, writer)` task per client.
- Server-level `dict[str, Table]` (table_key \u2192 `Table`), created lazily on first `join`.
- Each `Table` owns an `asyncio.Lock` guarding join/leave/bid/play/disconnect/reconnect mutations that span an `await` (defensive even though the event loop is single-threaded, per Devil's-Advocate review below), a `dict[Seat, ClientSession | None]` (each `ClientSession` now also tracks a `connected: bool` flag, A16), and a `Game | None`.
- The transport-agnostic `Game`/`RoundState` classes in `game.py` contain no I/O — they're pure state machines invoked synchronously by the connection handler, which then converts returned events into `broadcast`/`send_to` calls. No separate "game loop" task is needed since the game is fully turn-based/request-response driven.
- Client: two concurrent coroutines within each connection attempt — `receiver_loop` (reads socket, updates local `ClientState`, re-renders the live table view via `ui.py`/`rich.live.Live`) and a main loop that, when a `bid_request`/`play_request` arrives (signaled via `asyncio.Event`), prompts with a short numbered menu via `asyncio.to_thread(input, ...)` so the receiver keeps processing broadcasts (e.g., other players' bids, `connection_status` updates) without blocking. Wrapping this pair is a top-level **reconnect-supervisor loop** (A16) that restarts the connection with backoff if the socket drops mid-session.

## CLI Entry Points
- Server: `python -m coinche.server [--host 0.0.0.0] [--port 8765] [--target-score 1000]` (all optional, sane defaults).
- Client: `python -m coinche.client [--host HOST] [--port PORT] [--table KEY] [--name NAME]` — any omitted value falls back to an interactive `input()` prompt at startup (host/port/table key/player name, in that order).
- `pyproject.toml` additionally exposes `[project.scripts]` `coinche-server` / `coinche-client` for `pip install -e .` convenience (optional, does not replace the `-m` invocations).

## Pre-flight Devil's Advocate Review
- **Lock necessity**: since asyncio is single-threaded, a `Table`-level lock is technically redundant for atomicity between two `await`-free statements, but is kept because `add_player`/`remove_player`/`mark_disconnected`/`reconnect` mutation sequences do span `await` points (e.g., awaiting a write before updating state) — without the lock, two near-simultaneous `join`s for the last open seat (or a `join`/reconnect race) could both read stale state. Keeping the lock is the safer, low-cost choice.
- **Rich-markup injection**: called out explicitly as a required mitigation in step 8 (`ui.py`) — untrusted player names/chat text must never be embedded into a markup-format string; this is the one OWASP-relevant injection vector in an otherwise closed, LAN-style CLI tool. The keyboard-shortcut menu design (steps 8/9) also incidentally shrinks the free-text input surface (only `chat` and the initial player name remain free-form), a simplicity side-benefit, not a substitute for the `rich.text.Text(...)` mitigation.
- **Oversized/malformed input**: relying on asyncio's default 64 KiB `StreamReader` line limit is a reasonable, zero-extra-code DoS guard; `protocol.decode` additionally rejects non-JSON-object payloads and missing required fields per message type rather than trusting client-declared structure.
- **Table-key / name validation gap**: table keys and player names are not persisted or authenticated — any client that knows a table key can join if a seat is free; this matches the requested scope (no auth/persistence) but is worth restating so it isn't mistaken for a security boundary.
- **Tie/simultaneous target-score edge case (A13)** explicitly resolved to avoid an unhandled game-over branch; without it, `Game.submit_card`'s end-of-round check would have an ambiguous "both teams ≥ target" state with no defined winner.
- **Reconnection race — impersonation**: a new `join` only matches a *disconnected* seat by exact case-insensitive name; if the original player's socket is merely slow/not-yet-detected-as-dead rather than actually closed (a "still connected but silent" false-negative), a second client sending the same name collides with A15's `NAME_TAKEN` check (since the original seat is still `connected`), not treated as a reconnection — no impersonation window exists unless the server has already marked that seat `disconnected` (which only happens on an actual socket error/EOF, never speculatively).
- **Dead-write detection**: a broadcast/send_to write can fail (`ConnectionResetError`/`BrokenPipeError`) without the server having otherwise noticed the disconnect yet (e.g. the peer vanished but no read has been attempted); step 6 wraps `broadcast`/`send_to` in a `try/except` that calls `mark_disconnected` on a write failure so a "half-dead" socket is caught proactively rather than only being discovered on the next `readline()`.
- **Indefinite reservation of a disconnected seat**: since there is no timeout/forfeit logic (explicitly not requested), a permanently-gone player (crashed machine, never coming back) leaves that seat — and the whole table — stuck waiting forever with no way to free it up or end the game early. This is an accepted limitation of the requested scope (no bot/timeout logic); logged to discoveries.md as a candidate future improvement (e.g. an explicit "force-end"/"kick" admin action) rather than designed now.
- **Reconnect mid-bid vs. mid-trick**: `snapshot_for(seat)`/`resync` must branch on `phase` (`"bidding"` vs `"trick_play"`) since the relevant "current state" shape differs (bid history + legal bid actions vs. current trick + legal cards) — both are explicitly covered by the `resync` payload shape and by the `tests/test_game.py`/`tests/test_integration.py` coverage above so neither phase is an afterthought.
- **Incidental (undeclared) capot**: explicitly excluded from bonus scoring (A8) to avoid an ambiguous "should un-bid capot pay a bonus" rule not stated anywhere in context-map.md; normal tally applies, which is always at least as generous to the attacking team as their bid required.
- **Scaling limits**: acceptable for the stated scope (a handful of concurrent tables on one process, no persistence); not designed for hundreds of concurrent tables or multi-process/horizontal scaling — out of scope per discoveries.md.
- **What's deliberately NOT handled** (already logged in discoveries.md, not re-litigated here): bot/AI seat-filling, spectator mode, persistence/database, and any reconnection beyond a single running server process's lifetime (a server restart still loses all table state — reconnection per A16 only survives a *client-side* socket drop, not a server crash/restart). This plan does not touch those.
